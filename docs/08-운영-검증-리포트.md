## 🔴 필수 — 실제 사고로 이어지는 것

### R-1. 연차 일수 DOUBLE 타입 → DECIMAL로 변경
- **문제**: `base_days/use_days/bonus_days/advance_days/days/default_days` 전부 DOUBLE. 부동소수점은 0.5 연산이 누적되면 `7.999999…` 같은 오차가 생기고, `잔여 >= 신청일수` 검증이나 `== 0` 비교가 오작동한다. 연차는 **급여와 직결되는 수치**라 오차 자체가 신뢰 문제.
- **조치**: DB는 `DECIMAL(4,1)`, JPA는 `BigDecimal` (비교는 `compareTo`). 대안으로 "0.5일 = 1"로 정수 저장하는 방법도 있으나 표시 변환이 번거로움 → DECIMAL 추천.
- 적용 대상: users(4개 컬럼), leave_requests.days, welfare_policies.default_days, welfare_requests, leave_policy.annual_leave_days, leave_reset_history(5개 컬럼)

### R-2. @mlsoft.com 제한 — 이메일 문자열 검사만으로는 부족
- **문제**: 이전 프로젝트처럼 `email.endsWith("@mlsoft.com")` 검사만 하면, 도메인이 비슷한 외부 계정이나 검증 안 된 이메일에 취약. 또한 **퇴직자의 Workspace 계정이 살아있으면 OAuth는 성공**하므로 앱 차원 차단이 없으면 뚫린다.
- **조치** (3중 방어):
  1. **Google Cloud 콘솔에서 OAuth 앱을 "내부(Internal)"로 설정** — Workspace 조직 계정만 동의 화면 진입 가능. 회사 Workspace 관리자 계정으로 프로젝트를 만들어야 함 (이전 기획서의 "소장님께 요청" 항목이 바로 이것 — **도입 전 선결 조건**)
  2. 서버에서 ID 토큰의 `hd` claim == "mlsoft.com" **그리고** `email_verified == true` 검증 (hd 파라미터는 힌트일 뿐, 서버 검증이 본검증)
  3. 로그인 성공 핸들러에서 `is_active == false`(퇴직자)면 토큰 발급 거부 → `/login?error=retired`
- 명세 반영 위치: 01 §2-1 처리 규칙에 2)·3) 추가 필요.

### R-3. 최초 SYSTEM_ADMIN 부트스트랩 부재
- **문제**: 자동 가입은 전원 EMPLOYEE로 시작하고, 권한 변경은 SYSTEM_ADMIN만 가능 → **최초 관리자를 만들 방법이 설계에 없다** (닭과 달걀).
- **조치**: 환경변수 `ADMIN_EMAILS=someone@mlsoft.com` 지정 → 해당 이메일이 첫 로그인(자동 가입)할 때 SYSTEM_ADMIN 부여. DataInitializer 하드코딩보다 환경변수가 배포 유연성이 좋음.

### R-4. 이메일 발송 — 트랜잭션 분리 + 발송 상태 추적 없음
- **문제 1**: 승인 처리와 메일 발송이 한 트랜잭션이면 **메일 서버 장애 = 승인 실패**가 된다. 반대로 승인 커밋 후 메일만 실패하면 아무도 모른다.
- **문제 2**: `email_history`에 **발송 성공/실패 상태 컬럼이 없어** 실패를 추적·재발송할 수 없음. "메일이 안 왔어요"에 대응 불가.
- **조치**:
  - 발송은 `@Async` + 승인 커밋 후 실행 (`@TransactionalEventListener(AFTER_COMMIT)` 패턴 추천 — 승인 이벤트 발행 → 리스너가 발송)
  - `email_history`에 컬럼 추가: `status ENUM(PENDING/SENT/FAILED)`, `error_message VARCHAR(500)`, `sent_at DATETIME`
  - 실패 건 재발송: 스케줄러가 FAILED 재시도 (최대 N회) 또는 관리자 화면에서 수동 재발송
- **인프라 선결 조건**: noreply@mlsoft.com 계정 (Workspace 관리자에게 요청 — 이전 기획서 미해결 항목). Gmail SMTP는 Workspace 기준 **일 2,000건 한도** — 전 직원 일괄 리마인더 + 건별 알림이면 중소 규모에선 충분하지만, 인증은 앱 비밀번호보다 OAuth2 또는 SMTP relay 권장.

### R-5. 선차감 동시성 (레이스 컨디션)
- **문제**: 같은 유저가 두 탭에서 동시에 신청하면 둘 다 "잔여 충분" 검증을 통과해 **잔여 초과 신청**이 가능. 승인도 primary/sub가 동시에 처리하면 이중 처리 위험 (ALREADY_PROCESSED 검증은 있지만 읽기 시점이 같으면 뚫림).
- **조치**: users에 `@Version` 낙관적 락 (충돌 시 재시도 안내) — 내부 도구 트래픽엔 이걸로 충분. 승인 처리는 `SELECT … FOR UPDATE`(비관적 락) 또는 status 조건부 UPDATE(`WHERE status='PENDING'`)로 원자화.

## 🟡 도입 전 권장

### Y-1. 스케줄러를 "오늘이 입사일" 검색 → "last_reset_date 기준"으로
- 현재 명세: "매일 자정, 오늘이 입사일인 유저 검색". **그날 서버가 꺼져 있었으면 해당 유저 리셋이 1년 밀린다.**
- 권장: `last_reset_date + 1년 <= 오늘`인 유저 검색으로 변경 — 서버가 며칠 죽어도 재기동 시 자동 소급 처리(catch-up). 월차 적립도 동일 원리 적용. 생일 반차는 연 1회라 `생일이 오늘 이전 && 올해 미지급` 조건으로.

### Y-2. 온보딩 미완료 상태 정의
- 자동 가입 직후 hire_date/birth_day가 null인 유저가 존재. 이 상태에서 연차 신청·스케줄러 대상이 되면 오류.
- 권장: 온보딩 완료 전에는 온보딩 페이지 외 접근 차단 (`hire_date IS NULL` = 미완료 판별로 충분, 별도 플래그 불요). 스케줄러 쿼리에서 hire_date null 제외.

### Y-3. 부서 미배정 유저의 primary 승인자
- 팀장 퇴직 fallback(→SYSTEM_ADMIN)은 규칙이 있는데, **미배정 부서(leader 없음) 신입**의 신청은 규칙이 없음. 동일하게 SYSTEM_ADMIN fallback 명시 권장.

### Y-4. 개인정보 (개인정보보호법 관점)
- **사유 마스킹**(갭분석 A-8 🟡)을 내부 전문 사용 기준으로는 **필수로 격상 권장** — 캘린더에 전 직원 연차가 보이는 설계라 병가·경조 사유가 노출되면 민감정보 노출.
- 생일: 생일 반차 알림 외 화면 노출 최소화 (연도 숨김 등).
- 퇴직자 3년 보존은 근로기준법(임금대장 3년)과 정합 ✅. 3년 후 "관리자 처분 권한"에 **파기 절차**(개인정보 파기 의무)를 명시하면 완결.
- email_history.content에 개인정보가 담기므로 보존 기간을 퇴직자 정책과 맞출 것.

### Y-5. 인증 운영 디테일
- JWT 쿠키: `HttpOnly + Secure + SameSite=Lax` 명시 (Lax면 CSRF 방어 대부분 커버, CSRF 토큰 생략 가능한 근거가 됨)
- 내부망이어도 **HTTPS 필수** (쿠키 탈취 방지)
- 401 응답 시 프론트에서 로그인 페이지로 강제 리다이렉트 (이전 프로젝트 약점 — toast만 뜨고 방치)
- 토큰 24h + 갱신 없음은 내부 도구 기준 수용 가능. 즉시 차단이 필요한 퇴직 처리 시점엔 R-2의 is_active 검사가 로그인 재시도를 막아줌 (기존 토큰은 최대 24h 유효한 점만 인지)

### Y-6. 운영 준비
- dev/prod 프로필 분리, prod는 `ddl-auto: validate` (update 금지) + 스키마는 SQL/Flyway로 관리
- MySQL 일일 백업 (mysqldump cron — 전달받은참고자료 DEPLOYMENT.md 스크립트 재사용 가능)
- 공휴일 API(data.go.kr) 장애 대비: 연 1회 선조회 + DB 캐시 (이전 프로젝트 방식이면 충분)
