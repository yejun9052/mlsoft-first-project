# 01. 요구사항 & 기획

> 노션 기획 문서(2026-07 작성)를 정리한 것. MLsoft 휴가관리 시스템 재설계·재제작 기준 문서.

## 회고 (이전 프로젝트에서 얻은 교훈)

1. **AI 의존도가 너무 심했다** — 1달이라는 짧은 기간 탓에 AI에 의존. 이번엔 설계를 직접 주도한다.
2. **기능·DB 설계를 직접 하지 않았던 점이 후회됨** — 이번 프로젝트는 설계(요구사항/DB/API)를 먼저 직접 작성하고 구현에 들어간다.
3. 만들면서 "이런 부분이 있었으면" 했던 것들을 이번 설계에 반영 (이메일 알림, 퇴직자 보존, 서브 승인자 등).

> **작업 원칙**: AI는 보조 도구로 사용하되, 설계 결정과 세부 구현 이해는 본인이 주도한다.

## 요구사항 분석 (14개)

1. 로그인: OAuth 2.0, **@mlsoft.com만 허용**, 첫 로그인 시 자동 가입
2. 캘린더에서 공휴일·팀원 연차 조회
3. 연차 신청 시 사유 필수 (연차/반차 선택 가능)
4. 자신의 팀 정보 조회
5. 복리후생 페이지에서 연차 추가 신청
6. 총 연차 / 사용 개수 / 잔여 연차 조회
7. 신청 시 기본 승인자는 팀장, **서브 승인자 선택 가능**
8. 취소도 동일한 승인 구조
9. 신청 시 당사자·담당자에게 **메일 발송**
10. 승인 시 신청자·담당자·총관리자에게 메일 발송
11. 생일 반차 자동 추가 시 총관리자에게 메일
12. 30/60/90일·분기 단위로 기산일 임박 + 연차 잔여 사원에게 일괄 메일
13. 권한 체계: `SYSTEM_ADMIN`(총관리자) / `TEAM_LEADER`(팀장) / `EMPLOYEE`(사원)
14. 퇴직 사원 데이터 **3년 이상 보존** (근로기준법)

## 기능 명세

### 2-1 Google OAuth 2.0 로그인
- @mlsoft.com 계정만 로그인 가능, 첫 로그인 시 자동 가입 (권한 EMPLOYEE, 부서 미배정)
- **도메인 검증 3중 방어 (검증 R-2 — 명세만 확정, ⏳ 표시는 회사 계정 확보 후 적용)**
  - ① ⏳ Google Cloud OAuth 앱을 "내부(Internal)"로 설정 — **회사 Workspace 관리자 계정 확보 후 적용** (선결 조건, 그전까지 테스트 클라이언트 사용)
  - ② 서버에서 ID 토큰 `hd == "mlsoft.com"` + `email_verified == true` 검증 — 개발 중엔 이메일 도메인 검사로 대체하고, 운영 전환 시 hd 검증 활성화
  - ③ 퇴직자(is_active=false)는 OAuth 인증에 성공해도 토큰 발급 거부 → `/login?error=retired`
- **최초 관리자 부트스트랩 (검증 R-3)**: 환경변수 `ADMIN_EMAILS`에 지정된 이메일은 첫 로그인(자동 가입) 시 SYSTEM_ADMIN 부여 — 전원 EMPLOYEE로 시작하는 닭-달걀 문제 해결
- 최초 로그인 시 온보딩 페이지에서 **생일/입사일만 입력** → 완료 후 메인 이동
- **연차(base_days)는 유저가 입력하지 않는다** — 입력된 입사일 기준으로 연차 정책이 자동 계산 (위조 방지, 갭분석 C-1 반영)
- **온보딩 미완료 유저 (검증 Y-2)**: `hire_date IS NULL`이면 온보딩 페이지 외 접근 차단, 연차 신청·스케줄러 대상에서 제외
- 결과: JWT 토큰 발급 — 쿠키는 `HttpOnly + Secure + SameSite=Lax`, 운영은 HTTPS 필수. 401 응답 시 프론트는 로그인 페이지로 강제 리다이렉트 (검증 Y-5)

### 2-2 역할별 권한
- `SYSTEM_ADMIN`: 모든 권한 + 관리자 화면
- `TEAM_LEADER`: 팀원 연차 관리 + 자신을 서브 승인자로 지정한 신청 관리
- `EMPLOYEE`: 연차·반차·복리후생 신청/조회
- 권한 밖 페이지 접근 차단 (라우트 가드)

### 2-3 연차·반차 신청
- `ANNUAL`(1.0) / `HALF_AM` / `HALF_PM`(0.5) 선택, 날짜는 리스트로 저장
- 캘린더 날짜칸 클릭으로 선택
- 사유 필수
- **승인자 구조 (갭분석 B-3 반영)**
  - 기본 승인자(primary) = 소속 부서 팀장 (department.leader_id)
  - **부서 미배정 또는 팀장 공석이면 primary는 SYSTEM_ADMIN으로 fallback (검증 Y-3)**
  - **서브 승인자(sub) = 신청자가 재직 중(is_active=true)인 TEAM_LEADER·SYSTEM_ADMIN 중에서 직접 선택** (선택 사항 — 일반 EMPLOYEE는 후보 제외, 권한표와 일치)
  - 처리 방식은 **병렬 선착순**: primary/sub 중 먼저 처리한 1명의 결정으로 종료 (순차 결재 아님 — 팀장 부재 대비 목적)
- **동시성 방지 (검증 R-5)**: users 낙관적 락(@Version)으로 동시 신청에 의한 잔여 초과 차단, 승인·취소 처리는 status 조건부 갱신(`WHERE status='PENDING'`)으로 이중 처리 차단 (실패 시 ALREADY_PROCESSED)
- **잔여 검증 (갭분석 A-3 반영)**: 잔여 < 신청 일수면 거부가 기본. 단 시스템 설정 `advance_leave_enabled=true`면 당겨쓰기로 접수 (`users.advance_days` 누적, 다음 기산일에 정산)
- **기산일을 넘어가는 연차 신청은 다음 기산일에서 계산**
- 신청 시 당사자 + primary·sub 승인자에게 이메일
- 결과: `leave_requests`에 PENDING 저장 (**선차감**: use_days에 즉시 반영), 승인 권한자에게 요청 표시

### 2-3(b) 연차·반차 취소 (갭분석 A-2 반영)
- PENDING 상태: 날짜 무관 즉시 CANCELLED + 선차감분 복구
- APPROVED 상태:
  - **미래 날짜만 포함** → 즉시 CANCELLED + use_days 복구
  - **과거 날짜 포함(소급 취소)** → CANCEL_PENDING 전환, 승인자 승인 필요 (승인 시 CANCELLED+복구 / 거부 시 APPROVED 복원)
- 취소 사유는 `cancel_reason`에 저장, 처리 이력은 action_history에 기록

### 2-4 복리후생 신청·안내
- 기본 복리후생 안내 (페이지네이션)
- 안내 객체(행) 클릭 → 신청 폼에 자동 입력
- 승인 구조·이메일은 연차와 동일, `welfare_requests`에 저장

### 2-5 잔여 연차 확인 (연차 Status)
- 총 연차 / 사용 개수 / 잔여(총-사용) / 다음 기산일 차감 예정 표시
- 기산일은 **입사일 기준**
- **대기 중 연차도 use_days에 선차감, 거부되면 복구** ← 이전 프로젝트(승인 시 차감)와 다른 정책

### 2-5(b) 캘린더
- 공휴일 + 팀원 일정 + 전체 연차 표시, 월 이동 가능
- 날짜칸 클릭으로 연차 신청
- **사유 마스킹 (검증 Y-4, 🔴 확정)**: 타인의 연차는 이름·기간·종류만 표시. 사유는 **본인·해당 건 승인자·SYSTEM_ADMIN만 조회 가능** — 병가·경조 사유는 민감 개인정보

### 2-6 생일 반차
- 매일 자정 스케줄러가 생일인 유저 검색 → 0.5일 지급
- 지급 시 담당자와 당사자에게 메일

### 2-7 기산일 스케줄러
- 매일 자정 실행. **대상 검색은 "오늘이 입사일"이 아니라 `last_reset_date + 1년 <= 오늘`인 유저 (검증 Y-1)** — 자정에 서버가 꺼져 있던 날이 있어도 재기동 후 첫 실행에서 자동 소급(catch-up). 정책에 따라 총 연차 재지급
- 월차 적립·생일 반차도 동일 원리(마지막 지급 기록 기준)로 미실행분 자동 복구
- 기산일 기준 미래에 사용한 연차 → 사용 연차에 포함해서 계산
- **당겨쓴 연차(advance_days) 정산**: 새 base_days에서 차감 후 advance_days 초기화 (갭분석 A-1 반영)
- 기산일 이전 사용 신청은 새 총 연차/사용 연차에 영향 없음
- **1년 미만 신입 월차 적립 (갭분석 B-1 반영, 근로기준법 §60)**: 입사 1년 미만 유저는 매월 입사일(같은 날)마다 1일 적립, 최대 11일. 1주년 도래 시 월차 적립 중단 + 정책 연차로 전환
- **미사용 연차 소멸 (갭분석 B-2 반영)**: 기산일 리셋 시 남은 연차는 이월 없이 소멸. 리셋 전 상태(prev_base/prev_used/expired_days)는 `leave_reset_history`에 기록 — 디자인의 "소멸 예정 N일" 표시는 다음 기산일까지 남은 잔여를 의미

### 2-8 이메일 일괄 전송 (관리자)
- 기산일 임박 + 연차 잔여 사원 리스트 조회 (며칠 전부터 띄울지 설정 가능)
- 관리자가 메일 양식 직접 작성, 체크박스로 일부만 선택 발송 가능
- 30/60/90일·분기 기준 자동 발송 설정 가능
- 발송 내역 보존 (`email_history`)

**이메일 공통 처리 규칙 (검증 R-4 — 신청/승인/생일/리마인더 등 모든 발송에 적용)**
- 발송은 업무 트랜잭션과 분리: 커밋 후 비동기 발송 (`@Async` + `@TransactionalEventListener(AFTER_COMMIT)`) — **메일 서버 장애가 승인·신청 처리를 실패시키지 않는다**
- 모든 발송은 `email_history`에 기록: status `PENDING → SENT/FAILED`, 실패 시 error_message
- FAILED 건은 스케줄러 재시도(최대 3회) + 관리자 화면 수동 재발송
- 발송 계정: ⏳ noreply@mlsoft.com (회사에 요청 — 선결 조건). Gmail SMTP는 Workspace 기준 일 2,000건 한도

### 2-9 퇴직자 데이터 보존
- 퇴직 상태 변경은 SYSTEM_ADMIN만 가능
- 3년 이상 보존 후 처분 권한은 SYSTEM_ADMIN에게
- 퇴직자 별도 목록 조회
- **팀장 퇴직 시 결재 이관 (갭분석 B-4 반영)**: 해당 부서 `leader_id` 해제(null) → 후임 지정 전까지 기본 승인자는 SYSTEM_ADMIN으로 fallback. 퇴직 팀장 앞으로 걸려 있던 PENDING/CANCEL_PENDING 건도 동일하게 이관
- 퇴직자는 서브 승인자 선택 후보에서 제외 (is_active=true만 후보)
