# 02. DATABASE 설계

> 노션 설계 원문 기준. 13개 테이블. DB 네이밍: 테이블/컬럼 `snake_case`, JPA 필드는 `camelCase`.
> **일수(연차) 컬럼은 전부 `DECIMAL(4,1)` / JPA는 `BigDecimal`** — 부동소수점 오차 방지 (검증 R-1, 2026-07-02 확정).

## ERD 관계 요약

```
departments ─┬─< users (department_id)
             └── leader_id → users (팀장 명시)

users ─< leave_requests ─< leave_dates
                        └─< leave_action_history
users ─< welfare_requests (policy_id → welfare_policies)
                        └─< welfare_action_history
users ─< email_history (수신자/발신자)

leave_policy (근속년수별 연차), holidays (공휴일) : 독립 테이블
```

## 구현 시 검토할 점 (설계 리뷰 메모)

1. **created_at/update_at 타입**: 원문은 DATE — 감사 목적이면 DATETIME 권장. 결정 필요.
2. **users.role에 RETIRED 포함 vs is_active로 판별**: 둘 다 있음. RETIRED를 role로 둘지, is_active=false + 기존 role 유지로 갈지 정리 필요 (퇴직 전 직급 이력 보존 관점에선 후자 + 로그인 차단이 유리). ※ 승인자 후보·팀장 이관 규칙은 is_active 기준으로 확정됨(2026-07-02) — role 쪽만 결정 남음.
3. **대기 중 선차감 정책**: use_days에 PENDING 포함 → 화면 표시는 "사용(확정)"과 "대기"를 분리해서 보여줘야 혼란 없음.
4. **history의 user_id(신청자) 중복 보관**: 조인 없이 로그 조회하기 위한 반정규화 — 유지.
5. **인덱스**: users.email(UK), leave_dates.day, leave_requests.status, *_history.created_at, holidays.date — 전달받은참고자료의 인덱스 전략 참고.
6. **FK 삭제 정책**: 퇴직자 3년 보존 요구 때문에 CASCADE 금지, 사용자 삭제 대신 비활성화(소프트 삭제)로 통일.
7. **welfare_requests에 부여 일수 스냅샷 없음**: category/target/evidence_guide는 신청 시점 값을 복사하는데 **부여 일수는 컬럼이 없음** — 승인 후 정책의 default_days가 바뀌면 부여량 근거가 사라짐. `add_days DECIMAL(4,1)` 스냅샷 컬럼 추가 검토 필요.
