## A. 이전 프로젝트에는 있었는데 새 설계에서 빠진 것

| # | 우선순위 | 항목 | 이전 구현 | 의견 |
|---|---|---|---|---|
| A-1 | 🔴 | **당겨쓰기 저장 필드** | `users.advance_days` + `advance_leave_enabled` 설정 | 기능명세 2-7이 "당겨쓴 연차"를 언급하는데 **users 테이블에 저장할 컬럼이 없음**. `advance_days` 컬럼을 추가하거나, "use_days가 (base+bonus)를 초과한 만큼이 당겨쓴 것"이라는 계산 규칙을 명문화해야 함. 정산 명확성 면에서 컬럼 추가 추천 |
| A-2 | 🔴 | **취소 세부 규칙** | 미래 날짜만 → 즉시 취소 / 과거 포함 → CANCEL_PENDING(승인 필요) | status에 CANCEL_PENDING은 있지만 **언제 즉시 취소되고 언제 승인이 필요한지 규칙이 명세에 없음**. 이전 규칙(미래=즉시, 과거 포함=승인)이 합리적이므로 그대로 채택 추천 |
| A-3 | 🔴 | **잔여 부족 시 신청 규칙** | 부족 시 거부, 단 `advance_leave_enabled=true`면 당겨쓰기 허용 | 새 명세에 잔여 검증 규칙 없음. 선차감 정책과 묶어서: "잔여 < 신청 일수 → 거부(기본) / 당겨쓰기 허용 시 접수" 명시 필요 |
| A-4 | 🟡 | **관리자 구성원 관리** | 권한·부서 변경, 연차 직접 설정, 정책 기반 자동 부여(개별/일괄) | 디자인에 "구성원 관리" 메뉴가 이미 있는데 기능 명세가 없음. 2-10 자리에 명세 추가 필요 |
| A-5 | 🟡 | **팀(부서) 관리 CRUD** | 생성/수정/삭제 + 2단계 계층 + 트리 조회 | API 초안에 createDept만 있음. leader_id 지정 UI 포함해 명세 필요 (새 설계의 핵심인 "기본 승인자=팀장"이 leader_id에 의존하므로) |
| A-6 | 🟡 | **처리 로그 조회 화면** | 관리자 전체 로그 + 팀장 팀 로그 (액션 필터, 서버 페이징) | 테이블(action_history)은 설계했는데 조회 기능 명세가 없음. 결재 관리 화면의 "승인/반려" 탭이 이 역할 — 명세화 필요 |
| A-7 | 🟡 | **대시보드 명세** | 이번주 연차 현황, 공휴일 D-day, 내 대기 신청 | 디자인에 대시보드가 있는데 기능 명세가 없음 (2-5 잔여 확인이 일부만 커버). 디자인 기준으로 명세 작성 추천 |
| A-8 | 🟡 | **사유(reason) 마스킹** | EMPLOYEE가 타인 신청 조회 시 사유 숨김 | 캘린더에 팀원/전체 연차가 보이는 설계이므로 **개인 사유 프라이버시 보호 필요**. 유지 추천 |
| A-9 | 🟡 | **복리후생 취소 규칙** | PENDING 상태일 때만 취소 가능 | 본인의 이전 노션 메모에도 있던 규칙인데 새 명세에서 누락. 복리후생은 승인 시 연차가 가산되므로 승인 후 취소는 차감 롤백이 복잡 → PENDING만 취소 규칙 유지 추천 |
| A-10 | 🟡 | **신청 검증 (중복/휴일)** | 같은 날짜 중복 신청 거부, 주말·공휴일 클릭 차단 | 구현하면서 자연히 넣겠지만 명세에 명시해두면 테스트 기준이 됨 |
| A-11 | 🟢 | 회계연도 기준 기산일 옵션 | `reset_type` 설정 (입사일/회계연도 선택) | 새 요구사항이 "입사일 기준"으로 확정했으므로 **의도적 제외로 보는 게 맞음**. 단순화 찬성 — 이전에도 회계연도 모드는 검증 못 했음. 나중을 위해 config 확장 여지만 남기기 |
| A-12 | 🟢 | 사이드바 상세 정보 | 대기 중 일수, 다음 기산일 차감 예정 표시 | 새 디자인의 통계 스트립("소멸 예정", "결재 대기")이 대체 — 디자인 따라가면 됨 |
| A-13 | 🟢 | 연차 설정 잠금(isLocked) | 최초 설정 후 잠금, 관리자만 수정 | 새 설계는 "정책이 자동 지정"이므로 불필요. **대신 C-1 모순 해결 필요** |

## B. 둘 다 없지만 연차 관리에 꼭 필요하다고 보는 것

| # | 우선순위 | 항목 | 의견 |
|---|---|---|---|
| B-1 | 🔴 | **1년 미만 신입 월차 적립** | **근로기준법 제60조: 1년 미만 근로자는 1개월 개근 시 1일 유급휴가** (최대 11일). 이전 프로젝트 스케줄러에는 있었는데 새 명세에서 빠짐. 신입이 입사하면 연차 0인 상태가 되므로 반드시 필요. 기산일 스케줄러(2-7)에 "입사 1년 미만: 매월 입사일에 1일 적립" 추가 추천 |
| B-2 | 🔴 | **연차 소멸 처리** | **디자인에 "소멸 예정 2.0일"이 이미 그려져 있는데** 소멸 로직 명세가 없음. 기산일 리셋 시 미사용 연차를 소멸시킬지/이월할지, 소멸 이력을 어디 남길지(action_history? 별도?) 정의 필요. 2-8 이메일 촉진과 세트로 동작하는 핵심 규칙 |
| B-3 | 🔴 | **승인자 처리 규칙** | primary + sub 2단 구조인데 **"둘 중 아무나 한 명이 처리하면 종료"인지 "순차 결재"인지 미정**. 디자인 결재 라인(A→B 화살표)은 순차처럼 보이지만, 요구사항 7("팀장 부재 대비 서브 승인자")은 병렬(아무나)에 가까움. 추천: **병렬 — 둘 중 먼저 처리한 사람으로 종료** (구현 단순, 요구사항 의도 부합). 순차 결재는 실무에서 승인 지연의 주범 |
| B-4 | 🔴 | **팀장 퇴직/부재 시 결재 이관** | 퇴직자 기능(2-9)과 승인 구조가 만나는 지점. 팀장이 퇴직하면 department.leader_id 처리와 그 팀장 앞으로 걸린 PENDING 신청 처리 규칙 필요. 추천: 퇴직 처리 시 leader_id null → 기본 승인자 fallback을 SYSTEM_ADMIN으로 |
| B-5 | 🟡 | **퇴직 처리 시 진행 중 신청 정리** | 퇴직자로 전환할 때 본인의 PENDING 신청 자동 취소 + 미래 날짜 APPROVED 처리 규칙 필요 |
| B-6 | 🟡 | **이메일 자동 발송 설정 저장** | 2-8의 "30/60/90/분기 자동 발송 설정"과 "며칠 전부터 리스트 표시" 설정을 저장할 곳이 없음. `email_history`는 이력만. → **leave_policy_config(3-11 보류 중)를 이 시점에 확정**하는 게 자연스러움 (reminder_days, auto_send_cycle 등) |
| B-7 | 🟡 | **중복 처리 방지** | 승인 버튼 두 번 클릭, 두 승인자가 동시 처리 같은 경합 → "이미 처리된 신청입니다"(ALREADY_PROCESSED) 검증. 이전 프로젝트에 있던 예외 체계 재사용 |
| B-8 | 🟢 | 반차 0.5 단위·음수 검증 | days 계산 검증 (leave_dates 수 × 단가 = days 일치 확인 포함) |
| B-9 | 🟢 | 팀별 연차 사용 통계 | 전달받은참고자료(PTO 시스템)에 있던 개념. 관리자 대시보드 확장 아이템으로 2차 범위 추천 |
| B-10 | 🟢 | 인앱 알림함 | 이메일이 주 채널이므로 1차 범위에서 제외 추천. 나중에 email_history 재활용 가능 |

## C. 설계 내부 모순 / 결정 필요 (구현 전 정리)

| # | 항목 | 내용 | 추천 |
|---|---|---|---|
| C-1 | 온보딩 "연차 입력" 모순 | 2-1에 "연차 개수는 정책에 의해 기본 지정"과 온보딩에서 "연차 입력"이 공존 | **유저는 생일/입사일만 입력 → 입사일 기반으로 정책이 base_days 자동 계산**. 유저가 자기 연차를 입력하는 건 위조 여지가 있어 제외 추천 |
| C-2 | RETIRED role vs is_active | 두 방식 중복 (docs/02 §검토사항 2와 동일) | is_active=false + retired_at로 판별, role은 퇴직 전 직급 보존. 로그인 차단은 is_active 검사로 |
| C-3 | 선차감과 "사용 연차" 표시 | PENDING도 use_days에 포함되면 화면의 "사용 7일"에 대기분이 섞임 | DB는 선차감 유지하되, 화면은 "사용(확정) / 대기 중 / 잔여"를 분리 표시 (디자인 통계 스트립과 일치) |
| C-4 | 로그 테이블 분리 vs 통합 | 새 설계는 leave/welfare 히스토리 분리. 이전 노션 메모("enum으로 구분해 통합")와 방향이 다름 | **분리 유지 추천** (FK 무결성 명확). 관리자 로그 화면에서 통합해 보려면 두 API 병합 또는 UNION 뷰로 해결 |
| C-5 | holidays.month 컬럼 | 이전엔 year+month, 새 설계는 year만 | date에서 추출 가능하므로 새 설계대로 OK. 월별 조회 쿼리만 인덱스 고려 |

## 정리 — 설계에 반영할 최소 목록 (🔴만)

1. `users.advance_days` 컬럼 추가 (또는 초과 사용 계산 규칙 명문화) — A-1
2. 취소 규칙: 미래=즉시 / 과거 포함=CANCEL_PENDING — A-2
3. 잔여 부족 시 거부(기본) + 당겨쓰기 허용 여부 config — A-3
4. 1년 미만 월 1일 적립 (근로기준법) — B-1
5. 기산일 리셋 시 연차 소멸 규칙 + 소멸 이력 — B-2
6. 승인자 병렬 처리 규칙 (primary/sub 중 선착순 1명) — B-3
7. 팀장 퇴직 시 leader_id·대기 결재 이관 — B-4
8. 온보딩에서 유저 연차 입력 제거 (정책 자동 계산) — C-1
